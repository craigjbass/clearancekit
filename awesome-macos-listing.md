# ClearanceKit — Awesome macOS Listing

## Short Description

**ClearanceKit** — File-access control for macOS that protects sensitive user data (SSH keys, credentials, messages, browser cookies) from supply-chain attacks. Intercepts file-system operations via Endpoint Security and enforces code-signing-based policies, so a trojanised `npm` package or compromised build tool is denied access even if it runs under your user account. Policies bind to cryptographic signing identities rather than file hashes, so they survive software updates indefinitely. Ships with built-in presets for common apps (Signal, Notes, Contacts, Discord) and supports MDM-managed policy delivery.

## Comparison with BlockBlock and Santa

**BlockBlock** monitors persistence events — LaunchAgents, LoginItems, kernel extensions — and alerts you when something tries to install itself to survive a reboot. It answers the question *"is something trying to persist on my machine?"* but does not control which processes can read your files at runtime. A malicious npm postinstall script that exfiltrates `~/.ssh/id_rsa` and exits would never trigger BlockBlock, because it never persists. ClearanceKit operates at a fundamentally different layer: it intercepts every file open, rename, delete, and copy against protected paths and decides in real time whether the requesting process is authorised.

**Santa** overlaps more directly. Like ClearanceKit, Santa uses the Endpoint Security framework and can authorise both binary execution and file access, using hash-based or signing-certificate rules. Where they differ is in focus and policy model. Santa's primary strength is binary allowlisting/denylisting — controlling *what runs* — with file access rules as an additional capability. ClearanceKit is built entirely around file access: its policy language is oriented around protected path prefixes with per-rule process and ancestor constraints, and it ships with curated presets for common user-data targets (SSH keys, message databases, browser cookies). The two can be complementary: Santa controlling which binaries execute, ClearanceKit controlling what those binaries can read once running.

**Ancestry awareness** is where ClearanceKit diverges most sharply from both tools. Neither BlockBlock nor Santa model the process tree. ClearanceKit maintains a live in-memory forest of every running process with cached ancestor chains. Policy rules can specify both *who* is accessing a file and *who launched them* — e.g. "allow `git` to read `~/.ssh` only when its ancestor chain includes a process signed by my CI runner's team ID." This means the same binary can be granted or denied access depending on how it was invoked. A shell spawned by your terminal emulator gets a different policy outcome than the same shell spawned by a compromised build script. BlockBlock and Santa have no equivalent concept; they evaluate binaries in isolation without call-chain context.

**User control** also differs. BlockBlock is alert-driven — it pops a dialog and the user approves or denies in the moment, with no persistent policy language. Santa is admin-driven — rules are typically pushed via a sync server or MDM, and end users don't interact with policy directly. ClearanceKit sits between these: it provides a GUI that shows denied accesses in real time, lets users create signature-based allow rules from observed events with one click, and organises rules into named app protections that can be toggled per-app. It also supports MDM-managed policies for enterprise deployment, giving organisations the same central control model as Santa while preserving per-user visibility and override capability. Terminal denials are written directly to the user's TTY with a `SIGWINCH` to draw attention, so developers see access denials inline with their workflow rather than in a separate notification centre.
